┌─ DESIGN CHOICES CREATING BOTTLENECKS ──────────────────────────────────────┐
│                                                                             │
│ CHOICE 1: Sequential FSM Stage Handoff                                     │
│   ├─ Blocking: Stages wait for prior .done before starting                 │
│   ├─ Impact: 40 cycles idle (30% waste)                                    │
│   └─ Trade-off: Simpler FSM logic vs. high latency                         │
│                                                                             │
│ CHOICE 2: Fixed ARRAY_SIZE=4, Fixed DIM=8                                  │
│   ├─ Constraint: 4×4 PE grid cannot single-pass 8D embeddings              │
│   ├─ Impact: 2 matmul passes per token (14 cycles vs. optimal 7)          │
│   ├─ PE utilization: Only 25% active per cycle                            │
│   └─ Trade-off: Small area footprint vs. throughput                        │
│                                                                             │
│ CHOICE 3: Reactive DMA Loading (No Prefetch)                               │
│   ├─ Pattern: Compute blocks for weights → AXI handshake → data arrives   │
│   ├─ Impact: 8-15 cycle stalls per layer (hidden in small models)          │
│   ├─ Bonus: prefetch_engine.v ALREADY EXISTS but not wired!               │
│   └─ Trade-off: Simpler control vs. latency hiding                         │
│                                                                             │
│ CHOICE 4: Crude Softmax exp() Approximation                                │
│   ├─ Implementation: Shift-based piecewise (8 cases, ~5-10% error)        │
│   ├─ Impact: 5-10% numerical error, requires correction cycles             │
│   ├─ Reality: exp_lut_256 module exists but not used!                     │
│   └─ Trade-off: LUT area vs. accuracy/convergence                          │
│                                                                             │
│ CHOICE 5: Single KV Cache RAM                                              │
│   ├─ Access pattern: Read previous entries while writing new ones          │
│   ├─ Stall: Read-after-write dependency (must wait for write commit)       │
│   ├─ Impact: 2-3 cycles per token                                          │
│   └─ Trade-off: Memory area vs. throughput                                 │
│                                                                             │
│ CHOICE 6: Narrow Accumulators                                              │
│   ├─ Constraint: exp_val 8-bit, exp_sum 16-bit (limited dynamic range)    │
│   ├─ Impact: Saturation triggering extra correction (2-3 cycles)           │
│   ├─ Root: FPGA resource constraints                                       │
│   └─ Trade-off: Small storage vs. precision loss                           │
│                                                                             │
└────────────────────────────────────────────────────────────────────────────┘
