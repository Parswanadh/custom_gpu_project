## TOP-6 CRITICAL BOTTLENECKS (Ranked by Impact & Effort)

### 1. SEQUENTIAL PIPELINE STAGE HANDOFF ⭐⭐⭐ (30.8% of latency - 40 CYCLES)
**Impact:** Highest - 12-15 cycle reduction possible  
**Effort:** Medium - 5-7 days (architectural refactor)  
**Status:** Ready to fix (blocker for all other optimizations)

**Root Cause:**
- FSM has explicit wait states (S_ROPE_W, S_GQA_W, S_SOFTMAX_W, etc.)
- Each stage blocks until prior stage completes
- No inter-stage buffering (skid buffers)
- No ready/valid handshaking protocol

**Current Flow (BROKEN):**
\\\
[RoPE compute] WAIT(1 cycle) → [GQA compute] WAIT(1 cycle) → [Softmax] WAIT(1 cycle) ...
^-- Stall          ^-- Stall        ^-- Stall (30% of time = 40 cycles!)
\\\

**Fix Strategy:**
1. Remove wait states from FSM (S_ROPE_W, S_GQA_W, etc.)
2. Add .ready input to each stage module
3. Implement 1-entry skid buffers between all 5 inter-stage connections
4. Use ready/valid handshaking (combinational, low latency)

**Expected Outcome:** 115 cycles/token (-11% overall speedup)

**Files to Modify:**
- tl/integration/optimized_transformer_layer.v (FSM refactor)
- Create: tl/integration/skid_buffer.v (new pipelining module)

### 3. KV CACHE QUANTIZATION LOOP (11.5% of latency - 15 CYCLES)
**Impact:** High - 3-5 cycle reduction  
**Effort:** Medium (3-4 days, requires pipelining + reciprocal LUT)  
**Status:** Ready (all pieces exist)

**Root Cause:**
- Two sequential for-loops: min/max search → quantization
- Division operator (line 107) is ~3-4 cycle divider
- No parallelization between passes
- Scale factor computed as right-shift (crude)

**Current Implementation:**
`erilog
// LOOP 1: Find min/max (sequential, ~5 cycles)
for (i = 1; i < VEC_LEN; i++) { if (val < vmin) vmin = val; }

// LOOP 2: Quantize (sequential, ~10 cycles - division is slow!)
q_raw = (shifted_u + (scale_factor >> 1)) / scale_factor;
`

**Fix Strategy:**
1. Parallelize min/max tree (1 cycle, not 5)
2. Fuse quantization: Replace division with reciprocal LUT lookup
3. 4 parallel multiplications (fast) instead of 4 sequential divisions (slow)

**Expected Outcome:** 110 cycles/token (-15% speedup from #1+#2+#3)

**Files to Modify:**
- tl/memory/kv_cache_quantizer.v (fuse min/max + pipelining + recip LUT)

### 4. DMA ENGINE MEMORY STALL (3.8% latency, SCALES WITH MODEL SIZE)
**Impact:** Medium (6-11% speedup potential, hidden in small models)  
**Effort:** LOW (2-3 days - module ALREADY EXISTS, just needs wiring!)  
**Status:** Integration missing (module exists but not instantiated)

**Root Cause:**
- Weight loading is **REACTIVE** (compute requests → DMA starts → 2-4 cycle AXI handshake)
- Layer N+1 weights only load AFTER layer N finishes computing
- No prefetch orchestration
- **BUT:** prefetch_engine.v module exists (lines 31-68) with full ping-pong logic!

**Current Flow (INEFFICIENT):**
`
[Compute Layer N] blocks → [Wait for DMA Layer N+1] → [Compute N+1]
^-- Idle, no parallelism
`

**With Prefetch (OPTIMAL):**
`
[Compute Layer N] overlaps with [DMA Layer N+1]
^-- Both happen in parallel, 0 additional stall cycles
`

**Fix Strategy:**
1. Instantiate prefetch_engine in gpu_system_top_v2.v (10 lines)
2. Wire layer_done signal from compute completion
3. Coordinate DMA requests: prioritize compute, but allow prefetch in idle time
4. Connect dual-port weight buffer

**Expected Outcome:** Hide 8-15 cycles latency (6-11% speedup)
**Bonus:** Scales with model size (larger models = bigger win)

**Files to Modify:**
- tl/top/gpu_system_top_v2.v (instantiate + wire prefetch_engine)
- tl/control/layer_pipeline_controller.v (coordinate layer sequencing)

### 5. SYSTOLIC ARRAY SMALL-MATRIX STALL (6.2% of latency - 8 CYCLES)
**Impact:** Medium - 4-7 cycle reduction  
**Effort:** MEDIUM-HIGH (2-3 days, area cost ~4×)  
**Status:** Ready (parameter change or prefetch-based tiling)

**Root Cause:**
- Fixed ARRAY_SIZE=4 (4×4 PE grid)
- DIM=8 embeddings require 2 passes per matmul
- PE utilization: only 25% (1-2 PEs active per cycle)
- Total per-token matmul latency: 14 cycles (vs optimal 7 if single-pass)

**Current Architecture:**
`
ARRAY_SIZE = 4  →  Latency per pass = 4 + 4 - 1 = 7 cycles
DIM = 8         →  Requires 2 passes = 14 cycles total

PE Utilization: 8D input × 4-wide array = 25% (severely underutilized!)
`

**Option A: Increase array to 8×8 (HIGH COST, HIGH GAIN)**
- Area: ~4× (16 → 64 PEs)
- Latency: 14 cycles → 7 cycles (50% reduction!)
- Single-pass 8×8 matmul

**Option B: Implement prefetch-based weight tiling (LOW COST, MEDIUM GAIN)**
- Keep 4×4 array
- Load next pass's weights while computing current pass
- Effective throughput: 8 cycles (reduce pipeline fill overhead)

**Recommendation:** Option B first (lower cost), then Option A if area permits

### 6. SOFTMAX FIXED-POINT ACCUMULATION (1.9% - 2-3 CYCLES)
**Impact:** Low-Medium (1-2 cycle reduction + accuracy improvement)  
**Effort:** LOW (1-2 days, trivial changes)  
**Status:** Ready (just expand registers)

**Root Cause:**
- exp_val stored as 8-bit (limits to [0, 256])
- exp_sum stored as 16-bit (can overflow for 4-element vector: 4×255 = 1020)
- Saturation triggering extra correction cycles

**Fix Strategy:**
1. Expand exp_val to 16-bit (match LUT output width)
2. Expand exp_sum to 20-21 bit (prevent overflow)
3. Remove saturation handling code (not needed with wider accumulators)

**Cost:** Negligible (4 bytes register storage)
