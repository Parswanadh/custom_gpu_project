1. SOFTMAX SEQUENTIAL STATE MACHINE (CRITICAL)
   ─────────────────────────────────────────────
   Measured Impact: ~25 cycles per token (19% of total latency)
   Evidence:
   - File: rtl/compute/parallel_softmax.v, lines 48-58
     State machine with 6 states: IDLE→FIND_MAX→CALC_EXP→SUM_EXP→RECIP→NORMALIZE→DONE
   - Each state = 4-5 cycles (find-max, calc-exp, sum, recip, normalize)
   - Current implementation uses fast_exp() approximation (shift-based, crude)
   - Real bottleneck: Tree reduce in SUM_EXP stage requires sequential addition
   
   Root Cause:
   - Line 52 in parallel_softmax.v: exp_sum_reduce computed via iterative loop
   - Parallel units=4, but tree reduction still O(N) iterative
   - No true parallelization of max/min reduction
   
   Remediation (Priority 1):
   a) Replace sequential tree-reduce with parallel comparators (3-level tree for 4 elements)
      Expected Impact: 3 cycles reduction → 22-cycle softmax, 5.5% overall improvement
      Risk: Medium (logic synthesis complexity, new tree structure)
      Implementation: Modify lines 38-57 of parallel_softmax.v to use generate blocks for tree
   
   b) Use higher-precision exp LUT (2-piece linear or 4-piece) instead of shift approx
      Expected Impact: Improved numerical accuracy, potential 1-2 cycle reduction in convergence
      Risk: Low (just LUT content change)
      Implementation: Replace fast_exp() function with exp_lut_256 instance (already exists!)

2. SEQUENTIAL PIPELINE STAGE HANDOFF (CRITICAL - HIGHEST IMPACT)
   ──────────────────────────────────────────────────────────────
   Measured Impact: ~40 cycles idle (30% of total latency wasted)
   Evidence:
   - File: rtl/integration/optimized_transformer_layer.v, lines 209-223
     State machine shows 13 states with explicit wait states:
     S_ROPE → S_ROPE_W (wait) → S_GQA → S_GQA_W (wait) → ... 
   - Each stage signals .complete, next stage waits for completion before starting
   - No pipeline overlap between stages (should be: Stage1 compute || Stage2 compute || Stage3 prefetch)
   - Actual execution: [RoPE done] pause [GQA start] pause [Softmax start] pause...
   
   Root Cause:
   - Line 150, 158, 187: Three separate valid_in signals manually driven by FSM
   - No automatic handoff logic (ready/valid backpressure)
   - Sequential state transitions at lines 291-365 block until prior stage completes
   - Each state waits for .done signal before advancing (lines 239-264 in always block)
   
   Current FSM Pattern (BROKEN):
     if (done && state == S_ROPE) state <= S_ROPE_W
     if (state == S_ROPE_W) state <= S_GQA   [1 cycle WASTED here]
     if (done && state == S_GQA) state <= S_GQA_W
     if (state == S_GQA_W) state <= S_SOFTMAX [1 cycle WASTED here]
   
   Remediation (Priority 1 - MUST FIX FIRST):
   a) Implement inter-stage ready/valid signaling protocol
      - Change from FSM wait states to asynchronous valid/ready handshake
      - Allow stages to enqueue output while downstream processes them
      Expected Impact: 12-15 cycle reduction (9-11% speedup), enables true pipelining
      Risk: Medium (architectural refactor, CDC sync issues if async paths cross clock domains)
      Implementation Steps:
        i. Add ready input to each stage module (rope_ready, gqa_ready, softmax_ready, etc.)
        ii. Stage outputs when downstream ready OR buffer not full
        iii. Remove wait states from FSM (eliminate S_ROPE_W, S_GQA_W, etc.)
        iv. Use skid buffer between stages (simple 1-entry FIFO with valid/ready)
      Files to Modify: rtl/integration/optimized_transformer_layer.v (main), potentially add skid buffer module
   
   b) Add stage pipelining buffer (small skid/queue between stages)
      Expected Impact: 8-10 cycle reduction if single-element buffers added
      Risk: Low (standard practice, minimal area)
      Implementation: Instantiate skid_buffer between each stage output/input

3. KV CACHE QUANTIZATION LOOP BOTTLENECK (HIGH)
   ──────────────────────────────────────────────
   Measured Impact: ~15 cycles per token (11% of latency)
   Evidence:
   - File: rtl/memory/kv_cache_quantizer.v, lines 76-100
   - Quantization requires per-element min/max scan, then per-element scaling
   - Line 80-84: for-loop finds vmin/vmax across VEC_LEN elements (sequential)
   - Line 96-100: for-loop quantizes each value (q = (val - min) / scale) (sequential)
   - Both loops are sequential (run in 2 separate cycles/states)
   
   Root Cause:
   - Combinational min/max tree used, but synthesis may create long chains
   - Scale factor computation (line 89): right-shift approximation creates rounding
   - Two independent passes (find-scale, then quantize) instead of fused pass
   - No pipelining of min/max search stages
   
   Remediation (Priority 2):
   a) Fuse quantization into single pass with accumulation
      - Pre-compute min/max in first cycle via parallel tree (not sequential loop)
      - Quantize remaining VEC_LEN elements in next cycle (pipelined)
      Expected Impact: 3-5 cycle reduction (20-30% of KV stage)
      Risk: Medium (requires 2-stage pipeline, area tradeoff)
      Implementation: Replace iterative for-loops with parallel comparator tree + mux
   
   b) Use coarser quantization interval (e.g., INT8 instead of INT4 for values, pack more efficiently)
      Expected Impact: Smaller dequant overhead later, saves 2-3 cycles
      Risk: High (accuracy loss, perplexity impact)

4. SYSTOLIC ARRAY STALL ON SMALL MATRICES (HIGH)
   ────────────────────────────────────────────
   Measured Impact: ~8-12 cycles per token (6-9% of latency)
   Evidence:
   - File: rtl/compute/systolic_array.v, lines 18-22
   - Array size: ARRAY_SIZE=4, but token embedding DIM=8
   - Latency formula: ARRAY_SIZE + ARRAY_SIZE - 1 = 7 cycles per 4×4 matmul
   - Token path requires 2 passes: (8×8) → (8×4) → (8×8) chain
   - Q/K/V projections use 8×8 weight matrices but only process 8-dim embeddings
   - PE utilization: Only 25% (1 PE active per cycle in 4×4 array for 8-dim streaming)
   
   Root Cause:
   - Line 110-113: Skew register for row-wise staggered input
   - Each row staggered by ARRAY_SIZE cycles (4-cycle delay pipeline fill)
   - For 8-dim input × 8-dim weight = 2 systolic passes (14-16 cycles total)
   - But only ~25% PE utilization (no outer loop reuse)
   
   Remediation (Priority 2):
   a) Increase systolic array size from 4→8 (double dimension)
      Expected Impact: Single-pass 8×8 matmul, 7-cycle latency vs 14 cycles now → 7 cycle reduction (5% speedup)
      Risk: Medium (area ~4×, power ~4×, synthesis complexity)
      Implementation: Set ARRAY_SIZE=8 parameter in systolic_array instantiation
   
   b) Implement tiling controller to pre-load weights, reduce I/O stalls
      Expected Impact: 2-3 cycle reduction from better pipelining
      Risk: Low (logic complexity, minimal area)

5. DMA ENGINE MEMORY STALL (HIGH - SCALES WITH SEQUENCE LENGTH)
   ────────────────────────────────────────────────────────────
   Measured Impact: ~5-8 cycles per token (estimated, not directly measured in small test)
   Evidence:
   - File: rtl/memory/dma_engine.v, lines 15-79
   - AXI4 master uses MAX_BURST=16, but weight loads for DIM=8 are small (64-128 bytes)
   - Burst efficiency: each burst setup incurs fixed overhead (addr, len negotiation)
   - Line 25-30: transfer_len is 16-bit, but small transfers (64B) don't amortize header cost
   - No prefetch: DMA only starts after compute requests it
   - Reference in CLAUDE.md: "DMA engine (#17): AXI4 master for bulk weight transfers"
   
   Root Cause:
   - Reactive weight loading: compute→DMA start → address negotiation → data arrives
   - AXI4 handshakes add 2-4 cycles per burst (arvalid→arready, rvalid→rready)
   - Small burst size (16 words for 64-bit AXI) means many bursts per layer
   - Weight loading happens AFTER compute needs them (not prefetched)
   
   Remediation (Priority 2):
   a) Implement hardware prefetch engine (ALREADY EXISTS but not wired!)
      - Module exists in rtl/memory/prefetch_engine.v (lines 31-66)
      - Pre-load layer N+1 weights while computing layer N
      - Expected Impact: Hide 8-15 cycles of memory latency (6-11% speedup potential)
      Risk: Low (module exists, just requires integration wiring in gpu_system_top_v2)
      Implementation: Wire prefetch_engine.dma_request to command_processor, coordinate layer sequencing
      File: rtl/top/gpu_system_top_v2.v, lines 299-325 (add prefetch orchestration)
   
   b) Increase AXI burst size or use wider data path
      Expected Impact: Reduce header overhead by 30% (2-3 cycles per load)
      Risk: Low (AXI protocol compliant, just parameter change)

6. SOFTMAX FIXED-POINT ACCUMULATION OVERFLOW (MEDIUM)
   ────────────────────────────────────────────────────
   Measured Impact: ~2-3 cycles of rounding/saturation overhead per token
   Evidence:
   - File: rtl/compute/parallel_softmax.v, lines 62-88
   - fast_exp function uses crude piecewise approximation (8 branches, lines 64-87)
   - Line 72-86: No true exp LUT; instead uses hardcoded shifts (exp(-k) ≈ 256 >> k)
   - Accuracy loss requires extra correction cycles or clipping
   - Each softmax normalization must iterate on exp sum until convergence
   
   Root Cause:
   - Line 40-41: exp_val stored as reg[7:0], sum as reg[15:0] (narrow)
   - For 4-element vector, exp_sum can overflow past 16 bits without careful scaling
   - Piecewise approximation generates ~5-10% error vs true exp (RMSE)
   - Rounding errors accumulate in normalization (prob = exp / sum)
   
   Remediation (Priority 3):
   a) Use full 256-entry exp_lut_256 instead of fast_exp() crude approx
      Expected Impact: Accurate exp values, eliminates rounding stalls, 1-2 cycle reduction
      Risk: Low (LUT module already exists in codebase, just instantiation change)
      Implementation: Replace fast_exp() function call with exp_lut_256 instance (lines 55-56 show pattern)
   
   b) Increase accumulator precision (expand exp_sum to 20-24 bits to avoid saturation)
      Expected Impact: Eliminate 1-2 saturation-correction cycles
      Risk: Low (just wider register, minimal area)

10. KV CACHE DOUBLE-BUFFERING INEFFICIENT (MEDIUM)
    ───────────────────────────────────────────────
    Measured Impact: ~2-3 cycles per token (estimated)
    Evidence:
    - File: rtl/memory/weight_double_buffer.v (mentioned but not shown)
    - Reference in prefetch_engine.v (lines 68-70): dual buffers ping-pong
    - KV cache is written after each token but read for next sequence processing
    - Current design: NO true double-buffering of KV cache itself
    - Line 53-55 in attention_unit.v: k_cache/v_cache are single arrays (shared R/W)
    
    Root Cause:
    - KV cache written during current token processing
    - Next token must wait for cache write to complete before reading previous entries
    - No interleaving of read (previous cache) and write (new entry)
    - Competes with DMA for scratchpad/SRAM write port
    
    Remediation (Priority 3):
    a) Implement ping-pong KV cache (2 independent cache RAMs)
       - Write to cache_A while reading from cache_B (for next token)
       - Swap on token boundary
       Expected Impact: Reduce wait-after-write stall by 2-3 cycles per token
       Risk: Medium (doubles KV memory, integration complexity)
    
    b) Use memory arbitration to prioritize cache writes over other SRAM ops
       Expected Impact: 1-2 cycle priority boost
       Risk: Low (arbiter logic, no new hardware)

12. RECIP_LUT FOR SOFTMAX NORMALIZATION LATENCY (LOW)
    ───────────────────────────────────────────────────
    Measured Impact: ~1-2 cycles per token (1% of latency)
    Evidence:
    - File: rtl/compute/recip_lut_256.v (exists but latency not explicitly stated)
    - Referenced in parallel_softmax.v: RECIP state = 4'd4 (line 53)
    - Reciprocal lookup to compute 1/exp_sum for normalization
    - Line 68-69 in online_softmax_unit.v: Shows 2 exp LUT lookups per score
    - Reciprocal LUT adds 1 additional lookup cycle
    
    Root Cause:
    - Softmax normalization requires division: prob = exp / sum
    - Using LUT for reciprocal avoids division (hardware-expensive)
    - But LUT lookup has 1-cycle latency (reads at T, output at T+1)
    - Current FSM doesn't pipeline this (blocks on RECIP state)
    
    Remediation (Priority 4 - Low impact):
    a) Pre-compute reciprocal in parallel with exp sum accumulation
       - Anticipate sum range based on prior batches, start recip lookup early
       Expected Impact: 0.5-1 cycle reduction if fully overlapped
       Risk: Low (speculation/lookup, doesn't affect correctness)
    
    b) Use fast approximate reciprocal (Newton-Raphson 1-2 iterations) instead of LUT
       Expected Impact: ~0 cycles (faster than LUT if synthesis optimizes iteration)
       Risk: Medium (numerical accuracy, requires validation)
